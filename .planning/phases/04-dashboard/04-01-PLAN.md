<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser -> proxy.ts | Unauthenticated requests hit the proxy before reaching any page |
| Browser -> Server Actions | Server actions receive unchecked IDs from client components |
| Login form -> Supabase Auth | Credentials cross from browser to Supabase Auth API |
| Server -> Supabase DB | Service role key grants full DB access; only used server-side |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-01 | Spoofing | proxy.ts | mitigate | Verify session via `supabase.auth.getUser()` which makes a server-side call to Supabase Auth, not just decoding the JWT locally |
| T-04-02 | Spoofing | Server Actions | mitigate | Every server action independently verifies session via `createServerClient()` + `getUser()` — proxy alone is NOT a security boundary for mutations (per Next.js 16 docs) |
| T-04-03 | Information Disclosure | Supabase keys | mitigate | Service role key (`SUPABASE_SERVICE_ROLE_KEY`) only used in server components/actions via `@repo/db createClient()`. Anon key used in browser via `NEXT_PUBLIC_` prefix — this is safe as anon key is designed for client use with RLS |
| T-04-04 | Elevation of Privilege | Email allowlist | mitigate | Allowlist check runs server-side in a server action after auth succeeds. Uses `process.env.ALLOWED_EMAILS` (not NEXT_PUBLIC_), keeping the list hidden from clients. Non-allowed users are signed out immediately |
| T-04-05 | Tampering | Review approve/reject IDs | mitigate | Server actions validate the review item exists before mutation, and verify the session. IDs are UUIDs — no sequential enumeration risk |
| T-04-06 | Information Disclosure | Environment variables | accept | `NEXT_PUBLIC_SUPABASE_URL` and `NEXT_PUBLIC_SUPABASE_ANON_KEY` are intentionally exposed to the browser — these are public by design in Supabase's security model |
</threat_model>
