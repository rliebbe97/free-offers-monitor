# Plan 03-01: Offer Validation Cron

<objective>
Implement a daily offer validation system as a fourth concurrent loop in the worker process. The loop polls for active offers with `next_check_at <= now()`, performs HTTP liveness checks (HEAD with GET fallback), scans page text for dead signals via Cheerio, enforces a two-consecutive-failure expiry rule (at least 24 hours apart), treats 403/429 as WAF blocks (not failures), and writes every result to `verification_log`. Schema extended with `consecutive_failures` column on `offers` and a pg_cron entry for VAL-01.
</objective>

<context>
Key decisions from 03-CONTEXT.md that affect implementation:

- **D-01:** Validation runs as a fourth concurrent loop in the worker, matching the three-loop architecture from Phase 2.
- **D-02/D-03:** pg_cron is declared for VAL-01 compliance; actual execution is the TypeScript polling loop checking `next_check_at <= now()`.
- **D-04/D-05/D-06:** Dead signal phrases live in `apps/worker/src/validation/dead-signals.ts` as `readonly string[]`; matching is case-insensitive `toLowerCase().includes()` against `$('body').text()`.
- **D-07/D-08:** Normal recheck: 7 days + random jitter (0-6 hours). After first failure: 24 hours (no jitter). After WAF: 6 hours.
- **D-10/D-11/D-12/D-13:** HEAD first, GET fallback on 405/timeout. 10s timeout. User-Agent: `FreeOffersMonitor/1.0`. Follow redirects. 200-399=live, 403/429=WAF (not failure), 404/410/5xx=failed, network error=failed.
- **D-14/D-15/D-16:** Two consecutive failures → expire. A successful check resets to 0 failures and 7-day cycle.

From RESEARCH.md:
- `consecutive_failures` column needed on `offers` table (Option A — counter column, not log query).
- p-limit already in worker deps at 7.3.0 — use for concurrency control (5 concurrent requests).
- Cheerio NOT in worker deps — must be added.
- `verification_log` table already exists with all needed columns.
- `offers_next_check_active_idx` partial index already exists for efficient validation queries.
- Dead signal match on HTML that returns 200 counts as a failure (is_live = false).

Pitfalls:
- False dead signals from JS-rendered pages — two-failure rule mitigates.
- WAF/CDN blocking — 403/429 treated as check_failed, not incrementing consecutive_failures.
- Cheerio must be added to worker package.json before building.
- `raw_response` column is unbounded text — truncate to 2000 chars.
</context>

2. **`checkLiveness(url: string): Promise<LivenessResult>`** function:

   Implementation flow:
   a. Set common headers: `{ 'User-Agent': 'FreeOffersMonitor/1.0' }`
   b. Create abort signal: `AbortSignal.timeout(VALIDATION_REQUEST_TIMEOUT_MS)`
   c. **Try HEAD request:**
      - `fetch(url, { method: 'HEAD', headers, signal, redirect: 'follow' })`
      - On success with status 200-399: return `{ isLive: true, isWaf: false, httpStatus: status, deadSignals: [], rawText: null }`
      - On 403 or 429: return `{ isLive: false, isWaf: true, httpStatus: status, deadSignals: [], rawText: null }`
      - On 404, 410, or 5xx (>=500): fall through to GET (HEAD may not be supported properly)
      - On 405: fall through to GET
      - On network error (catch block): fall through to GET
   d. **GET fallback (new abort signal since HEAD may have consumed time):**
      - `fetch(url, { method: 'GET', headers, signal: AbortSignal.timeout(VALIDATION_REQUEST_TIMEOUT_MS), redirect: 'follow' })`
      - On network error: return `{ isLive: false, isWaf: false, httpStatus: null, deadSignals: [], rawText: null }`
      - On 403 or 429: return `{ isLive: false, isWaf: true, httpStatus: status, deadSignals: [], rawText: null }`
      - On 404, 410, or 5xx: return `{ isLive: false, isWaf: false, httpStatus: status, deadSignals: [], rawText: null }`
      - On 200-399: read body as text, run dead signal scan
   e. **Dead signal scan (only on successful GET with body):**
      - `import * as cheerio from 'cheerio'`
      - `const $ = cheerio.load(bodyText)`
      - `const pageText = $('body').text().toLowerCase()`
      - `const deadSignals = DEAD_SIGNALS.filter(phrase => pageText.includes(phrase))`
      - `const rawText = pageText.slice(0, VALIDATION_RAW_RESPONSE_MAX_CHARS)`
      - If deadSignals.length > 0: return `{ isLive: false, isWaf: false, httpStatus: status, deadSignals, rawText }`
      - If no dead signals: return `{ isLive: true, isWaf: false, httpStatus: status, deadSignals: [], rawText }`

   Key details:
   - HEAD success with 2xx-3xx returns immediately (no body to scan — that's expected)
   - Dead signal scan only runs on GET 200-399 responses (we have a body to analyze)
   - `isWaf: true` signals the caller NOT to increment consecutive_failures
   - All network errors caught and returned as `{ httpStatus: null }` — never throws

5. **`runValidationCycle(db)` function** (internal):
   ```typescript
   async function runValidationCycle(db: DbClient): Promise<void> {
     const { data: dueOffers, error: queryError } = await db
       .from('offers')
       .select('id, destination_url, consecutive_failures')
       .eq('status', 'active')
       .lte('next_check_at', new Date().toISOString());

     if (queryError) {
       throw new Error(`Failed to fetch due offers: ${queryError.message}`);
     }

     const offers = dueOffers ?? [];

     if (offers.length === 0) return;

     logger.info('validation_cycle_start', { due_count: offers.length });

     const limit = pLimit(VALIDATION_CONCURRENT_LIMIT);

     await Promise.all(
       offers.map((offer) => limit(() => validateOffer(db, offer)))
     );

     logger.info('validation_cycle_complete', { processed: offers.length });
   }
   ```

6. **`runValidationLoop(db, shutdown)` function** (exported):
   ```typescript
   export async function runValidationLoop(
     db: DbClient,
     shutdown: { stop: boolean },
   ): Promise<void> {
     while (!shutdown.stop) {
       const cycleStart = Date.now();

       try {
         await runValidationCycle(db);
       } catch (err) {
         logger.error('validation_loop_error', { error: String(err) });
       }

       const elapsed = Date.now() - cycleStart;
       const remaining = Math.max(0, VALIDATION_POLL_INTERVAL_MS - elapsed);

       if (!shutdown.stop && remaining > 0) {
         await sleep(remaining);
       }
     }

     logger.info('validation_loop_stopped');
   }
   ```

- p-limit(5) caps concurrent HTTP requests
- Each offer failure is caught individually — does not crash the cycle
- verification_log written for EVERY outcome (live, dead, WAF, network error)
- Two consecutive failures trigger `status: 'expired'`
- WAF does not increment consecutive_failures
- Successful check resets consecutive_failures to 0
