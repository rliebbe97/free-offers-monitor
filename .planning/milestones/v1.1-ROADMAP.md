# Milestone v1.1: Forum Adapters

**Status:** ✅ SHIPPED 2026-04-22
**Phases:** 5-6
**Total Plans:** 10

## Overview

Expanded ingestion beyond Reddit with a TheBump community adapter and reusable adapter infrastructure. Built shared scraping utilities, a BaseForumAdapter abstract class with template-method pagination, the concrete TheBumpAdapter with Cheerio scraping, and migrated the production pipeline to a type-agnostic source dispatch factory.

## Milestone Summary

**Key Decisions:**

- Decision: Template-method pattern for BaseForumAdapter (Rationale: Subclasses only provide selectors; base owns pagination, rate limiting, challenge detection)
- Decision: p-throttle for time-windowed rate limiting (Rationale: Complements existing p-retry and p-limit without overlap)
- Decision: Synthetic HTML fixtures over live snapshots (Rationale: Deterministic tests; Vanilla Forums structure verified against live site)
- Decision: Dedup cosine section advisory-only in eval runner (Rationale: Requires VOYAGE_API_KEY; shouldn't block Tier 1 eval)

**Issues Resolved:**

- vi.clearAllMocks vs vi.resetAllMocks: Vitest 3.x requires resetAllMocks to drain mockResolvedValueOnce queue
- ESM .js extension consistency across all cross-file imports
- THEBUMP_BASE_URL uses `??` fallback to avoid breaking vitest when env var not set

**Issues Deferred:**

- Live TheBump scrape validation (Cloudflare may block)
- pnpm eval execution with live API keys
- DB migration application to Supabase
- Live worker run with type='bump' source row

**Technical Debt Incurred:**

- `thebump_pagination_stop` log name in BaseForumAdapter is TheBump-specific but used generically — future adapters inherit misleading event name
- `fetchActiveSources` has no source status/active filter — no way to disable a source without deleting the row
